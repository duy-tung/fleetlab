# Heterogeneous placement (FL-T007) — reduced scope, honestly

**Scope reduction, invoked as pre-decided, not cut under pressure.** This
repo's entire measured evidence corpus covers exactly **one** hardware
bucket: the CPU-only mock/llama.cpp loopback family used throughout
FL-T004/FL-T005/FL-T006/FL-T008 (`docs/risks.md` FL-L2 — the GPU corpus
never materialized; program GPU budget ~$150–250 total). Per the
pre-decided kill rule (`docs/risks.md`, "FL-T007 heterogeneous-placement
depth is reducible to two hardware profiles") and the provenance mandate
("placement reasoning is restricted to hardware actually covered by
measured profiles — no extrapolation to unmeasured GPUs", `docs/non-
goals.md`), this task implements the full placement **mechanism** — tested,
working code, `fleetlab/placement/` — and runs it over exactly two hardware
candidates:

- **(a) measured:** `mock-loopback-cpu-dev__mock-8b__gateway-mock-flags-v1-
  conncap2` — the FL-T004 fitted, **G8-within-error** capacity profile.
- **(b) demonstration only:** `hardware-a10g-g5-xlarge` — the
  `serving-contracts` example NVIDIA A10G GPU profile, `basis: source-
  reported` (VRAM, price) / `basis: assumed` (its capacity figure, borrowed
  from a vendor illustrative fixture).

**Recommendations over unmeasured hardware are demonstrations of mechanism,
not advice.** This is enforced in code, not prose: every hardware candidate
carries a `PlacementVerdict` whose `is_recommendation` field is computed
from the candidate's own `basis` (`fleetlab/placement/model.py`,
`placement_verdict`), and a dedicated test
(`tests/placement/test_model.py::
test_placement_verdict_construction_enforces_its_own_invariant`) proves a
`PlacementVerdict` cannot be hand-constructed to claim otherwise. In this
report: **(a) `is_recommendation=true`; (b) `is_recommendation=false,
demonstration_only=true`.**

Code: `fleetlab/placement/model.py` (six mechanisms, pure/closed-form, no
RNG), `fleetlab/placement/build_placement_report.py` (wiring). Tests:
`tests/placement/` (36 tests: known-answer + the two sanity invariants named
in `docs/tasks.md`, asserted directly against code behavior). Raw output
(seeded — no RNG anywhere in this task, so nothing to seed; deterministic by
construction): `reports/scenarios/placement.json`, regenerated by
`python3 -m fleetlab.placement.build_placement_report`.

## 1. The two sanity invariants (enforced in code)

1. **Never place a model that doesn't fit VRAM (generalized to VRAM-or-RAM
   memory).** `memory_fit()` raises `MemoryCapacityUnknownError` — it does
   not return `fits=True` — whenever either the hardware's memory capacity
   or the model's weight size was never recorded. This bites immediately on
   candidate (a): **the measured CPU bucket's host RAM was never measured
   anywhere in this program's evidence** (it is a local-dev-container; only
   its fitted capacity_rps is measured). Pairing it with the one real model
   this program actually served on a CPU host (Qwen2.5-1.5B-Instruct,
   GGUF Q4_K_M, `weights_size_gb=1.117` **measured**), the memory-fit check
   returns `insufficient-data`, not a fabricated pass — an honest PENDING,
   the same discipline as FL-T004's memory-profile PENDING and FL-T002's
   "no hardware-profile example for the CPU-only host" deviation. On
   candidate (b), Llama-3.1-8B (`weights_size_gb=16.1`, source-reported)
   fits the A10G's 24 GB VRAM with 7.9 GB headroom (67.1% utilization) —
   this succeeds only because (b) is source-reported/assumed data, still
   never presented as a recommendation (§0 above).
2. **Never recommend unmeasured hardware.** `PlacementVerdict.
   is_recommendation` is `True` iff `hardware_basis == "measured"` —
   computed, not asserted by the caller. Candidate (a): `true`. Candidate
   (b): `false`, `demonstration_only: true`.

## 3. Throughput/cost ranking

Only one candidate carries both a `capacity_rps` and a `usd_per_hour`: the
GPU demo bucket (capacity 7.41 rps, an **assumed** figure borrowed from the
vendor's illustrative `goodput.requests_per_second_meeting_slo` fixture —
`vendor/.../examples/benchmark/result.json`, itself labeled "all values
illustrative, no measurement claims"; price $1.006/hr on-demand,
source-reported, as of 2026-07-10). The measured CPU bucket has a real
**measured** `capacity_rps` (26.157 ± 1.233 rps) but **no cost figure at
all** — a local-dev-container has no cloud billing, and this report does
not fabricate a $0 price to force a comparison. **With only one candidate
carrying both figures, the ranking mechanism runs but has nothing real to
rank against — this is itself the reduced-scope finding**, not a bug in the
mechanism (`tests/placement/test_model.py` exercises the ranking mechanism
itself against synthetic two-candidate fixtures, where it does discriminate
correctly).

## 4. Cold-start weighting

Neither of the two ranked candidates above has a measured cold-start
figure (the mock backend's manifests all declare `warm_up.policy: "none"`;
no GPU model-load timing exists anywhere in this program's evidence). The
mechanism is instead demonstrated directly against the one real cold-start
measurement this program has — FL-T005's `MEASURED_COLD_START` (llama.cpp/
Qwen2.5-1.5B, `inference-lab/evidence/i3`, same 4-vCPU CPU-only host family)
— against the real `bursty` workload's 15 s burst-phase duration as the
reaction window:

| Regime | Cold-start seconds (measured) | Penalty factor |
|---|---|---|
| warm (OS page cache hit) | 1.94 | **1.0** (comfortably inside the window) |
| cold (page cache evicted) | 91.34 | **0.164** (overruns the window ~6x) |

This is the same warm/cold gap FL-T005's cold-start-headroom report found
(`reports/cold-start-headroom.md`), now expressed as a placement-ranking
penalty rather than a backlog/drain-time figure — the mechanism, not a new
measurement.

## 5. Failover headroom

Composes `fleetlab.dynamics.headroom` (reused, not reimplemented) with the
measured CPU bucket's fitted per-replica capacity: 2 replicas against the
real `bursty` workload's 20 rps peak.

| | Value |
|---|---|
| Per-replica capacity | 26.157 rps (measured) |
| Full fleet (2 replicas) | 52.314 rps |
| N−1 (1 replica) | 26.157 rps |
| Peak offered (real `bursty` workload) | 20 rps |
| Deficit | **0** — no headroom deficit |

Matches FL-T005's published finding exactly (`reports/cold-start-
headroom.md`): this fitted capacity comfortably covers the real workload's
peak even with one replica down. No GPU failover scenario is computed: the
GPU demo bucket's "capacity" is a single borrowed illustrative figure, not
a real per-replica number worth composing into a failure scenario.

## 7. Workload affinity — hypothesis 5

Planning-prompt hypothesis 5: *"On heterogeneous fleets, workload affinity
(long-context vs short-chat) changes optimal placement even when per-GPU
cost/throughput ratios are equal."* Tested here on the **same**
hardware+model pairing (GPU demo × Llama-3.1-8B — identical throughput/cost
ranking by construction, since it's the same pairing) across two workloads:
`chat-short` (real, IB-T003 canonical: mean 200 input + 145.5 output ≈ 345.5
tokens) and `rag-long-in` (vendored non-normative `serving-contracts`
example fixture, owned by inferbench: mean 8000 input + 250 output = 8250
tokens).

| Workload | Mean context tokens | Required KV memory (1 request) | Concurrency headroom (requests) |
|---|---|---|---|
| `chat-short` | 345.5 | 0.045 GB | **174.5** |
| `rag-long-in` | 8250 | 1.081 GB | **7.3** |

Both fit a single typical request comfortably (the GPU's 7.9 GB post-weight
headroom covers even the long-context request's 1.08 GB). But the
**concurrency headroom collapses by ~24x** (174.5 → 7.3 concurrent
requests) purely from the workload's context-length profile, with the
hardware+model pairing and its raw throughput/cost ranking held completely
fixed — direct, quantified support for hypothesis 5: workload affinity
changes what "the same" placement can actually sustain, a dimension the
throughput/cost ranking in §3 does not see at all.

## 8. Limitations

- **This is the central limitation of the whole report, stated once and
  inherited everywhere:** candidate (b) is not a placement recommendation.
  Its VRAM and price are real, source-reported facts about a real A10G GPU;
  its capacity figure is a borrowed vendor-illustrative number
  (`basis: assumed`) used only so the ranking/fragmentation/affinity
  mechanisms have a second candidate to run against. Nothing here is
  evidence about what this program's mock backend, or any real GPU, would
  actually deliver if deployed.
- The measured CPU bucket's host memory (RAM) was never recorded anywhere
  in this program's evidence — the memory-fit and fragmentation mechanisms
  cannot evaluate it at all for that bucket (§2, §6), not a weakness of the
  code but a genuine data gap (see `profiles/examples/README.md`'s existing
  "no hardware-profile example for the CPU-only host" deviation — the same
  schema gap resurfaces here).
- The Qwen2.5-1.5B model paired with the measured CPU bucket (§2, §6) was
  never actually served by the specific mock-loopback engine that produced
  that bucket's fitted capacity — it is the best-available real
  (measured-weights) model to pair with a CPU-class hardware candidate in
  this demonstration, not a claim that this exact pairing was measured
  end-to-end.
- The cold-start weighting demo (§4) borrows a measurement from a different
  engine (llama.cpp/Qwen) than either ranked candidate — the only real
  cold-start figure available anywhere in this program — clearly scoped as
  its own exhibit, not attributed to the mock backend or the GPU.
- Simulation ≠ production (program-wide pledge); every number here is
  either measured on a CPU-only mock backend or explicitly labeled
  source-reported/assumed for a real GPU this program never benchmarked.

