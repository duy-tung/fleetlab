**Model (one free parameter, closed-form):**
`achieved_rps(offered) = min(offered, capacity_rps)`.
`capacity_rps` is solved exactly from the training point
(`capacity_rps = achieved_rps` of that point) — never via an iterative
optimizer (`fleetlab/fitting/capacity.py`).

**Overfitting guard:** 1 fitted parameter, 1 training data point, always.
This is structural, not a review-time check: `fit_capacity` takes exactly
the training points named in the `HoldoutSplit` and nothing else
(`fleetlab/fitting/holdout.py`).

## 3. G8 acceptance: MISS, documented as a limitation

**Stated error:** the corpus provides no independent basis to declare a
target error bound in advance (no prior fleetlab profile, no vendor SLA);
the working bar used here is "prediction error should be within a small
multiple of the measurement-noise floor." **The miss is 4–9x the
measurement-noise floor in every direction and every config** — this is a
genuine model-specification limitation, not sampling noise, and is reported
as such rather than absorbed into a wider, unjustified error bar.

**Root cause (analytic, not hand-waved):** the one-parameter capacity-clamp
model assumes a fixed capacity `C` independent of offered load. The real
system's admission-controlled queue evidently has some capacity dependence
on offered load itself (e.g. more contention/scheduling overhead at 189 rps
offered than at 37.8 rps offered, even though both arms use the same 4-vCPU
mock-loopback host and the same gateway build) — the two single-point
capacity estimates (33.16 vs 37.93 rps for `admission-sane-v1`; 31.01 vs
37.35 rps for `admission-sane-v1b`) differ by exactly the reported error,
by construction. A one-parameter model cannot capture that dependence; a
two-parameter model would have zero holdout data left to validate against
with only two real points per config. This is recorded as the honest limit
of what this corpus supports, not patched by adding an unvalidated second
parameter.

**Latency profile: PENDING**, not fitted at all — every real point in both
configs already sits at or above its own single-point capacity estimate, so
the queueing-blowup latency model's one parameter has no training point
where it is defined. See `docs/notes/fitting-method.md` §4 for the exact
missing-data requirement (one below-capacity calibration run per config).

**Memory profile: still PENDING**, carried forward from FL-T003
unchanged — no isolated KV-cache-memory measurement exists anywhere in the
currently available evidence (`docs/notes/model-validation.md` §5.2).

## 5. What would close the gap

1. **Capacity model:** a genuine rate sweep (3+ offered rates spanning
   underloaded -> at-capacity -> overloaded) for `admission-sane-v1` and
   `admission-sane-v1b` would let a 2+ parameter model be fit with a real
   holdout set left over, and would directly test whether capacity is
   offered-load-dependent (§3's root-cause hypothesis).
2. **Latency model:** one below-capacity calibration run per config (offered
   rate ~15–25 rps, clearly below the ~31–38 rps fitted capacities).
3. **Memory profile:** an isolated KV-cache-memory measurement (see
   `docs/notes/model-validation.md` §5.2's precise requirement — unchanged).
4. **Real hardware:** everything fitted here is the mock backend. A
   real-GPU or real-CPU-llama.cpp rate sweep (not a single point) is needed
   before any profile in `profiles/fitted/` could inform a real capacity
   recommendation (FL-T009) about real hardware.

None of these are assumed or fabricated in the meantime — every gap above
is a stated, addressable data requirement, and everything published in
`profiles/fitted/*.json` is exactly what could be honestly computed from
what exists today.
