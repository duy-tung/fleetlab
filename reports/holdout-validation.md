## 1. What was fitted

profile — every other (hardware, model, engine-config) combination in the

All fitted configs are the **mock backend** — profiled honestly as a
"hardware" config for **loop-mechanics purposes only** (per this task's

**Models (one free parameter each, closed-form):**
`achieved_rps(offered) = min(offered, capacity_rps)` for throughput
(`capacity_rps` by exact weighted least squares — for a single clamped
training point this reduces to `C = achieved` exactly, so the ib-t010
profiles are unchanged by the LSQ upgrade; ADR-0002 addendum), and
`e2e_p50(offered) = l0 * C / (C - offered)` for latency (defined only below
`C`; fitted only where sub-capacity training points exist).

**Overfitting guard:** 1 fitted parameter per model against 1–4 training
points. Structural, not a review-time check: `fit_capacity`/`fit_latency`
take exactly the training points named in the `HoldoutSplit` and nothing
else (`fleetlab/fitting/holdout.py`).

**Latency root cause (analytic):** the implied `l0` falls monotonically
across training points (55 ms at the lowest rate → 18 ms near the knee) —
the target's real latency is *additive* (fixed ~60 ms base service time +
queueing delay) while the one-parameter model is *multiplicative*, forcing
predicted latency → `l0` as load → 0 and under-predicting the empty-system
latency. A two-parameter additive form is now validatable with holdout left
over (6 points), deferred to a reviewed follow-up rather than swapped in
mid-task (`docs/notes/fitting-method.md` §4).

## 3. G8 acceptance for the ib-t010 configs: MISS, documented as a limitation

**Stated error:** the corpus provides no independent basis to declare a
target error bound in advance (no prior fleetlab profile, no vendor SLA);
the working bar used here is "prediction error should be within a small
multiple of the measurement-noise floor." **For the two ib-t010 configs the
miss is 4–9x the measurement-noise floor in every direction** — a genuine
model-specification limitation, not sampling noise, reported as such rather
than absorbed into a wider, unjustified error bar. (The ib-t008 sweep
config, by contrast, meets the same bar at ≈1.05x — §2b.)

**Root cause (analytic, not hand-waved):** the one-parameter capacity-clamp
model assumes a fixed capacity `C` independent of offered load. The real
system's admission-controlled queue evidently has some capacity dependence
on offered load itself (e.g. more contention/scheduling overhead at 189 rps
offered than at 37.8 rps offered, even though both arms use the same 4-vCPU
mock-loopback host and the same gateway build) — the two single-point
capacity estimates (33.16 vs 37.93 rps for `admission-sane-v1`; 31.01 vs
37.35 rps for `admission-sane-v1b`) differ by exactly the reported error,
by construction. A one-parameter model cannot capture that dependence; a
two-parameter model would have zero holdout data left to validate against
with only two real points per config. This is recorded as the honest limit
of what this corpus supports, not patched by adding an unvalidated second
parameter.

**Latency profile: PENDING for E2/E2b** — every real point in both configs
already sits at or above its own single-point capacity estimate, so the
queueing-blowup latency model's one parameter has no training point where
it is defined for them. (FITTED for the ib-t008 sweep config, §2b, where
sub-capacity points exist.) See `docs/notes/fitting-method.md` §4 for the
E2/E2b missing-data requirement (one below-capacity calibration run per
config).

**Memory profile: still PENDING**, carried forward from FL-T003
unchanged — no isolated KV-cache-memory measurement exists anywhere in the
currently available evidence (`docs/notes/model-validation.md` §5.2).

## 5. What would close the remaining gaps

1. **Capacity model, E2/E2b configs:** a rate sweep like ib-t008's for
   `admission-sane-v1` and `admission-sane-v1b` specifically (the existing
   sweep is a different gateway config and a different mock latency
   profile — profiles are never pooled across configs). The ib-t008 result
   (§2b) demonstrates the mechanism works when the data exists.
2. **Latency model, E2/E2b configs:** one below-capacity calibration run
   per config (offered rate ~15–25 rps, clearly below the ~31–38 rps fitted
   capacities). For the sweep config, the next step is instead the reviewed
   two-parameter additive form (§2b's root cause).
3. **Memory profile:** an isolated KV-cache-memory measurement (see
   `docs/notes/model-validation.md` §5.2's precise requirement — unchanged;
   nothing in ib-t008 measures memory either).
4. **Real hardware:** everything fitted here is the mock backend (the sweep
   config additionally behind a disclosed client-side concurrency cap). A
   real-GPU or real-CPU-llama.cpp rate sweep is needed before any profile
   in `profiles/fitted/` could inform a real capacity recommendation
   (FL-T009) about real hardware.

None of these are assumed or fabricated in the meantime — every gap above
is a stated, addressable data requirement, and everything published in
`profiles/fitted/*.json` is exactly what could be honestly computed from
what exists today.
