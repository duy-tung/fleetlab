# Cost model + sensitivity (FL-T008)

**MODEL DEMONSTRATION, not a real cost claim.** The only fitted
capacity/latency profile in this repo with a G8-within-error outcome
(`mock-loopback-cpu-dev__mock-8b__gateway-mock-flags-v1-conncap2`,
`docs/notes/fitting-method.md`) was measured on a **CPU-only mock backend**
— there is no GPU anywhere in this repo's evidence corpus, and no real
billing record to price it against. Every dollar figure below combines
that fitted profile's real, measured capacity/latency numbers with
`profiles/examples/cost-g5-xlarge-ondemand.json` — a real NVIDIA A10G GPU's
example on-demand/spot pricing, vendored from `serving-contracts`'s
reference examples — purely to demonstrate the cost-model mechanism end to
end. **This is a hardware/config mismatch, stated here and in every
generated artifact**: nothing in this report is a claim about what this
mock backend, or any real GPU, actually costs to run this workload.

Code: `fleetlab/cost/` (`model.py` — parameterized, hardware-agnostic
closed-form arithmetic; `build_cost_report.py` — the demonstration wiring).
Tests: `tests/cost/` (18 tests, known-answer + determinism). Raw output:
`reports/scenarios/cost-model.json`, regenerated by
`python3 -m fleetlab.cost.build_cost_report` (deterministic — pure
arithmetic, no RNG, nothing to seed).

**The SLO/latency-model mismatch, stated once here and inherited by every
number below:** the fitted latency model predicts e2e **p50**
(`docs/notes/fitting-method.md`); the SLO objective above is defined on
p95. Inverting a p50 model against a p95 threshold to get "goodput at SLO"
is a stated approximation, not a rigorous claim (`fleetlab/cost/
model.py`'s module docstring). At this SLO's threshold (10 s, very loose
relative to this system's ~39 ms near-empty latency) the approximation's
error is immaterial — goodput-at-SLO comes out at 99.6% of raw capacity
either way — but it would matter more at a tight SLO (§3).

## 2. Cost at SLO (demonstration)

| Pricing | $/hr | Goodput at SLO (e2e p95 < 10s, 99.6% of capacity) | Cost / successful request | Cost / 1M tokens |
|---|---|---|---|---|
| On-demand | 1.006 | 26.056 rps | $0.0000107 | **$0.181** |
| Spot (assumed) | 0.42 | 26.056 rps | $0.0000045 | **$0.0756** |

Goodput is identical across pricing models (price does not change what the
fitted profile predicts the system can deliver); spot pricing is ~58%
cheaper per token here, exactly the ratio of the two rates (0.42/1.006 =
0.417), since cost scales linearly in price at fixed goodput.

## 3. Sensitivity: price vs. SLO vs. load

Deterministic closed-form sweep (`reports/scenarios/cost-model.json`
`sensitivity.points`, 60 points: 4 price multipliers × 5 SLO thresholds × 3
load fractions). Two slices, holding load fraction at 1.0 (running exactly
at each SLO's own goodput ceiling):

**Price sensitivity** (SLO fixed at the loose 10 s threshold): cost per 1M
tokens scales exactly linearly with price, as expected —

| Price × on-demand | Cost / 1M tokens |
|---|---|
| 0.5× ($0.503/hr) | $0.0906 |
| 1.0× ($1.006/hr) | $0.1811 |
| 1.5× ($1.509/hr) | $0.2717 |
| 2.0× ($2.012/hr) | $0.3623 |

A 4× price range produces exactly a 4× cost range.

**SLO sensitivity** (price fixed at on-demand): tightening the SLO toward
the fitted `l0` (the model's near-empty-system latency, ≈0.0387 s) shrinks
goodput and drives cost per token up sharply and **nonlinearly**:

| SLO threshold | Goodput at SLO | Cost / 1M tokens |
|---|---|---|
| 10 s (99.6% of capacity) | 26.056 rps | $0.1811 |
| 1 s (96.1%) | 25.146 rps | $0.1877 |
| 0.2 s (80.7%) | 21.102 rps | $0.2237 |
| 0.1 s (61.3%) | 16.046 rps | $0.2941 |
| 0.0406 s (l0 × 1.05, 4.8%) | 1.246 rps | **$3.7892** |

A price range of 4× (§ above) produces a 4× cost range; tightening the SLO
from loose to just above `l0` produces a **~21×** cost range — five times
larger than the full price sensitivity band. This directly supports
`docs/experiments.md` hypothesis 4 ("cost per 1M tokens at SLO is most
sensitive to goodput near the saturation knee, not to raw GPU price"),
asserted as an invariant in `tests/cost/
test_build_cost_report.py::test_sensitivity_slo_tightening_dominates_price_range`
(not just read off the table once): as offered load is pushed toward the
model's own latency floor, the denominator of "cost per token"
(throughput) collapses much faster than any realistic price negotiation
could offset it. The practical reading: for this system, a capacity
planner gains far more by relaxing an unnecessarily tight SLO (or adding
capacity so the same SLO is met further from the knee) than by chasing a
better GPU price.

## 4. Recompute check against the corpus

**Every real benchmark-result in this repo's evidence carries
`cost: null`** (all 10 result files under `tests/fitting/fixtures/real/`
and `tests/golden/fixtures/real/` — checked directly, not assumed) — so
there is no real measured cost figure in this repo's actual corpus to
recompute against. The one file in this repo with a populated `cost` block
is `vendor/serving-contracts-v0.2.0/examples/benchmark/result.json`, an
illustrative fixture (its own `comparability_note`: "all values
illustrative, no measurement claims") referencing the same
`cost-g5-xlarge-ondemand` profile used throughout this report. This is the
one recompute check the corpus actually supports, and it is exact enough to
be reassuring about the arithmetic:

| | Value |
|---|---|
| Stated `output_tokens_per_second` | 1430.5 |
| Stated `per_million_output_tokens_usd` | 0.196 |
| Recomputed (`cost_per_1e6_tokens_usd(1.006, 1430.5)`) | 0.19535 |
| Relative error | **-0.33%** |

The other cost figure in that file, `per_million_tokens_usd` (0.161, priced
on *total* input+output tokens/s), **cannot be independently recomputed**
from that same file: `benchmark-result.schema.json`'s `throughput` block
carries `output_tokens_per_second` but no total-token-rate field — a
schema-coverage gap, recorded here rather than backed into with an assumed
input-token rate.

## 5. Limitations

- Simulation ≠ production; the priced hardware (A10G GPU) is not the
  hardware the capacity/latency numbers were measured on (a CPU-only mock
  backend) — this is a demonstration of the cost-model mechanism, not a
  cost prediction for any real deployment.
- The fitted latency model predicts p50; the SLO objective used is
  defined on p95 (§1) — a stated approximation.
- Spot pricing is an explicit planning assumption (~40% of on-demand), not
  a quoted price, and does not price in preemption risk.
- All prices are dated 2026-07-10 and are volatile ecosystem facts —
  re-verify at use time (program-wide rule).
- Tokens-per-request (59.21) is measured from the mock backend's own
  synthetic request generator, not from any real chat/RAG workload's token
  distribution — it is the right figure for internal consistency with the
  capacity/latency profile it is paired with, not a general "typical
  request" estimate.

## 6. Reproduction

```
python3 -m fleetlab.cost.build_cost_report
pytest tests/cost -q
```

Deterministic: no RNG anywhere in this task (`fleetlab/cost/model.py` is

test_build_report_is_deterministic` asserts byte-identical JSON across two
calls in the same process. Every input file's sha256 is embedded in
`reports/scenarios/cost-model.json`.
