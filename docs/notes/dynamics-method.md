# Dynamics method note (FL-T005)

Companion to `docs/notes/model-validation.md` (FL-T003) and
`docs/notes/fitting-method.md` (FL-T004): what `fleetlab/dynamics/`
implements, how it was validated, and exactly which delay parameters are
measured vs assumed.

## 1. The discrete-event core

`fleetlab/dynamics/simulator.py` implements the standard FCFS `G/G/c` queue
simulation algorithm (Law & Kelton): a shared FIFO queue feeds `c` identical
servers via two small heaps (server free-times; outstanding departures). It
is a pure function of its `arrivals`/`service_times` input arrays — no RNG,
no wall-clock — so any caller-supplied stochastic input (e.g. exponential
service times from a seeded `numpy.random.Generator`) is the only source of
randomness, exactly mirroring `fleetlab.models`'s determinism rule.

**Validated against five independent analytic limits**
(`tests/dynamics/test_simulator.py`), per ADR-0001's rule that every
discrete-event scenario must reproduce its analytic anchor:

1. **Deterministic single- and multi-server drain time** (no RNG at all): N
   already-queued jobs, fixed service time T, c servers -> drains in exactly
   `ceil(N/c) * T`.
2. **λ<μ (M/M/1):** simulated mean wait converges to the closed-form `ρ/(μ-λ)`
   within 5% at `N=200,000` samples.
3. **λ>μ:** in-system count grows linearly at rate `(λ-μ)`, measured between
   two instants while arrivals are still flowing (not during the post-
   arrival drain-down tail, which is a different, decreasing regime).
4. **M/M/c at low utilization:** simulated mean wait matches the closed-form
   Erlang-C formula within 10% (`c=3`, `ρ=0.3`, `N=150,000`).
5. **Burst decay:** a `bursty`-shaped phased-Poisson scenario, provisioned
   above its long-run average rate, returns to a low in-system count before
   every subsequent burst (asserted structurally, not just eyeballed).

Admission-controlled shedding (a `capacity` cap on total in-system count) is
exercised separately and asserted to never exceed the cap.

## 2. Delay parameters: what's measured, what's assumed

| Parameter | Value | Basis | Source |
|---|---|---|---|
| Warm model-load time | 1.94 s (mean of 6 real runs) | **measured** | `inference-lab/evidence/i3/logs/llama-server-*.log`, `MM.SS.mmm.uuu` elapsed-time field, reverse-engineered and cross-checked this session (see `fleetlab/dynamics/cold_start.py`) |
| Cold model-load time | 91.34 s (mean of 2 real runs) | **measured** | same logs, the two runs where the OS page cache was evicted |
| Scale-up lag | 11.94 s (= 10s assumed sched + 1.94s measured warm-load) | **mixed — assumed dominant term is flagged** | `fleetlab/dynamics/scaling.py`; the 10s pod-scheduling constant has no measured basis anywhere in this program's evidence (checked: zero matches for `scale up`/`replica`/`cooldown`/`autoscal*` across the full corpus) |
| Scale-down lag | 30 s | **assumed** | common Kubernetes graceful-drain/termination-grace default; not measured |

The llama.cpp log timestamp format (`MM.SS.mmm.uuu`, minutes uncapped rather
than rolling into an hours field) was not documented anywhere found this
session; it was reverse-engineered by testing candidate formats against an
independent check — the real wall-clock span between the first and last
`events.jsonl` timestamps of the `chat-short-cpu-gw` run (~600–610 s) against
that log's own final line (`10.09.254.080` = 10 min 9.254080 s = 609.25 s
under this format) — and confirmed exactly consistent. All eight per-log
deltas this note and `fleetlab/dynamics/cold_start.py` cite were verified
independently in Python (not by eye) against this parsing rule.

## 3. Scenario outputs

`reports/scenarios/bursty-queue-growth.json` and `reports/scenarios/
cold-start-headroom.json`, regenerated by `python3 -m
fleetlab.dynamics.build_scenarios` (deterministic; the queue-growth
scenario is seeded, `seed=20260711`, and embeds the input workload file's
sha256; the headroom scenario is pure closed-form arithmetic over measured/
fitted inputs, `seed=null`, and embeds both its input file digests). Full
narrative: `reports/cold-start-headroom.md` (one of the five required
reports).
